KeyTakeaways

1.Ton 的历史和发展

TON 发展历程曲折，从 2018 年创建， 19 年 ICO 在 20 年被 SEC 打回，以 new TON 的形式启动后， 22 年 TON 基金会重启生态建设；Telegram bot 生态热点多，Telegram Wallet 和 TON Space 预期 11 月开全量，带来了 TON 近期市值增长（$ 3 B-$ 7 B，+ 292% ）估值水平高（mcap / TVL = 654）；目前筹码结构集中，基金会大力推动 OTC 交易；目前已转入 POS，每年 0.6% 通胀率，当前质押可获得约 3.73% 的 APY；

Telegram 用户情况：MAU 8 亿；TON 活跃用户 3.5 M，预期覆盖 30% Telegram 用户，Target 2028 500 M 用户；用户覆盖非中美用户（印度 1 亿 DAU 3 亿 MAU，俄罗斯、印尼和马来市占率超 60% ）；Telegram 用户标签，关注新闻、娱乐和教育、耦合推特生态（59.5% 用户也使用 Twitter）、粘性高（日均使用时长 4 h）；

TON 基础设施机会：底层开发语言为  FUNC （难度大，类似 c++），近期官方提供了可以使用脚本语言的编译器 TACT ，效率有折损；TON 生态开发者早期来自 Durov 创建的 VKontakte 前员工，目前应用层核心 dev 来自韩国和俄罗斯；

2.Ton 的框架和技术

TON 架构：使用了自适应无限分片多链架构，包括 masterchain（协调核心链）、workchain（工作链，为分片链的集合）、shardchain（分片链，对工作链进行进一步分割，提升处理效率）。每个账户都作为一条 shardchain，可自适应组合成更大的 shardchain 以适应交易需求。TON 通过自适应无限分片、异步信息传递和特殊的共识机制，构建了一个高效且安全的区块链网络。

TON 实现了异步信息传递和超立方体路由机制。系统中的每个交易仅在单个智能合约上执行，通过异步消息来进行通信。消息从一条分片链经过超立方体路由至目标分片链，也提供了快速路由，即预处理及路由过程中的消息。

TON 共识机制：采用了 PoS 和 BFT 的混合方式，网络中含有验证节点、Fisherman 和校对人等角色。验证节点通过质押，每月选举一次。Fisherman 和校对人分别负责检测验证节点的工作和推荐新的验证节点。

MEV 和 闪电贷攻击等问题还没出现。每个节点通过相邻节点传递信息，采用不同的信息路径，并无 mempool 结构。通过“快路由”机制，已被相邻分片链传递的交易会从输出队列中删除，防止双花情况发生。

TON 优势和问题：Solana 和以太坊具有区块确认时间快，交易时间短，最大分片数量多以及分片通信速度快等优势。每一个智能合约需要支付其自身的资源成本，包括计算、存储和网络传输等，优秀的是在 TON 余额用尽时，智能合约会被自动删除，从而避免区块链数据膨胀。TON 的异步设计也造成智能合约间调用广泛的一致性和原子性更难维护，使应用开发及维护工作更加复杂。

3.TON 原生组件

TON P2P network：DHT 定位 + ADNL 通信的网络传输协议；

TON DNS ：分布式域名系统，用户和开发者可用，单价高昂；

TON storage：采用 BT 技术的类 dropbox 存储方案，“去中心化的 Amazon S 3 ”，开发者可在 storage 上部署合约，用户端使用复杂；

TON Proxy ：隐藏 TON 节点 IP 地址的网络匿名层，基本可用；

TON payments：有两个方向，@Wallet 面向 web2 用户，仅能收付款，目前推出 web3 向的 TON space，聚合支付、社交等生态；

Telegram Apps Center ：供 dapp 开发者无缝开发集成上链；

4.Telegram 生态和 Ton 生态

目前 Telegram 生态形态「Bots+API+钱包托管」，数据好但对 TON 公链没有实际支持；

项目方为什么要去 TON 生态？Telegram 内 Apps Center 倒流、TON 原生支付、telegram 社群推广、基金会 Grant 和投资 、解除 API 限制等均可助力项目发展；对应的体验是：Instant launch、User friendly interface、Push notifications、Social and game mechanics、Seamless Web3 integration。

Telegram 为什么要扶持 TON ？Telegram 有变现需求，上市预期未知、收入结构单一，而 TON 可能是商业化最好的途径之一；

当前 TON 生态应用主要「Bots+API+钱包托管+TON 代币映射」的形式，完全采用 FUNC 开发的应用较少、基建相对难用，因此 TONfura 等开发者工具中间件兴起；

官方已收录 551 个应用，包括质押、钱包、公共设施、NFT、社交、游戏、去中心化交易所等，DEX、借贷、游戏数量较多，生态丰富；

5.投资参考思路

看好官方扶持的基础设施：TON 组件可用但体验有待提升，围绕基建进行组合、优化体验，如一站式部署 infra、动态 site 搜索引擎、dashboard 和 aggregator 等方向；

看好小程序应用：重视小程序的多样性和广度，偏向 mass adoption 的 web 2.5 ，产品体验性和安全性大于 crypto 正统性。Ton space 和 telegtam 入口端能吸引大量web2流量，应用侧有流量来源、应用场景、社交关系、支付网关和用户触达的需求，在这五个方向可以投资一些强技术/生态背景的项目；

谨慎看好 DeFi、MEV 和 ZK ：目前生态最需要的 DeFi 产品偏向稳定收益池和交易基础设施，而不是复杂的 DeFi 产品创新；在传统 crypto 方向，考虑 MEV、ZK 和 bridge 为时尚早，当前首要目的是“造”更多资产，搭建稳定产品链；短期内不看好 Layer 2 和 zk 方向；

其他赛道方向：Dev Tools 对团队技术要求高，需保持与官方的紧密沟通+官方组件耦合，难度较大；Social Payment 和游戏+社交图谱的绑定方向可以考虑与代币经济结合的小游戏，或解决创作者与用户交互的 Social platform。

开发者情况

开发语言

底层语言：有两种，一个是 FUNC，类似 C++，难度很高；另一个是偏底层的 Fift，包含 TVM 和 Fift 汇编指令，难度更高；

优化的编译语言：目前推出了高级的编译语言 TACT，开发难度可降低 40% ，有 type scripe 开发经验上手会很快，对标开发者情况是大厂高级开发员能很快上手。其特点如下：

1. 语法友好：Tact 语法受 JavaScript、Rust 和 Swift 的启发，在代数数据类型和编译时执行等方面较为友好；

2. 面向 Actor：Tact 是专为 TON actor 模型设计的强类型消息强制参与者之间的通信契约，提供与 TL-B 方案兼容的代数数据类型；

3. Gas 控制：Tact 通过 Gas commit 和编译器进行 verify，Gas 成本要么是静态限制，要么在运行时会显式检查。

开发者开发时需注意如下几点（来自 Beosin 安全团队）：

1. FunC 与 Tact 均为静态类型语言，开发者需要非常清楚地知道其代码中变量存储的数据类型；

2. TON 区块链不包含 revert 消息，开发者需要考虑好代码运行的结束路径；

3. TON 区块链有多个交易阶段（computational phase、actions phase 和 bounce phase），开发者在测试代码时需要注意交易执行目前位于第几阶段；

4. TON 区块链是异步的，开发者需注意处理失败的调用信息；

5. 对于外部消息，开发者需要注意重放攻击的风险。可以通过设置计数器或是标识符来避免重放攻击。

3.Ton 的框架和技术

自适应无限分片多链的架构

TON 的分片是自下而上的，即先将账户链分组为碎片链，然后在碎片链中进行交互。与其他区块链分片方案不同的是，TON 让多条链并行处理交易，被称为“blockchain of blockchains”。

TON 的架构包括三个重要组成部分：Masterchain、Workchain 和 Shardchain。Masterchain 是协调中枢，包含协议的参数、Validators 集合和对应的 share，以及当前工作的 Workchains 和下属 Shardchains。下层的链会将其最新的区块哈希提交至 Masterchain，以确保能够确定最新状态。

1. Masterchain：仅有唯一一条，包含了协议的参数、Validators 集合以及对应的 share，当前工作的 Workchains 及下属 Shardchains。下层的链会将其最新的区块哈希提交至 Masterchain，这样当下层链需要发生跨链的消息读取时，能够确定最新状态。

2. Workchain：是一个虚拟概念，作为 Shardchains 的集合存在，系统最高可容纳 2 ^ 32 条 Workchains。每条 Workchain 在满足互操作性标准的前提下，可以灵活地自定义规则，例如地址格式、交易类型、原生代币、智能合约虚拟机等。

3. Shardchain：为提升处理效率，每个 Workchain 会进一步分割为分片 Shardchains，最高可达 2 ^ 60 （即分片能在负载变高时自动分为两个，并在负载下降时合二为一）。Shardchain 遵守 Workchain 定下的规则，将工作水平分摊到所有的 Shardchain 中，每条 Shardchain 仅为所有账户集合中的一部分服务。

信息传递机制 - 异步 message 传递 + message 超立方体路由机制

Message 简介：TON 中各节点传递的称为“message”，这和它的开发语言有关（FunC 的 send_raw_message函数）。基本逻辑是参与者处理传入 message，更改其内部状态，并最终生成出站 message；

超立方体路由简介：无需考虑系统中分片链的总数，使一个分片链区块中创建的消息迅速传递和处理至目标分片链的下一个区块中。

异步的 message 传递

在异步系统如 TON 中，每笔交易仅在单个智能合约上执行，通过消息进行通信。这与同步的区块链，如以太坊，有所不同。在以太坊中，一笔交易可能影响和改变多个合约的状态。

为了实现此类系统，TON 引入了逻辑时间或 Lamport 时间的概念，用于处理事件的顺序。每条消息都有其逻辑时间，这可以用来确定哪个事件先发生，从而让验证器能决定优先处理的事务。操作逻辑保证了消息的逻辑时间严格按照先后顺序执行，并且有多个消息时，逻辑时间较低的消息会被更早处理。

Message 超立方体路由机制

传统超立方体路由包括三种链之间的消息传递：同一个工作链的分片链之间、不同工作链的分片链之间、以及主链和其他工作链之间。每个链都仅连接到与其分片标识符不同的一个十六进制数的分片，形成“超立方体”图形。整个分片链网络组成一个超立方体网络，需要的路由条数为 log 16(N)，能支持在 4 个节点的条件下支持百万级的分片链。

https://youtu.be/V4 TXwxDS 0 Ss? si=_OvNMQJDntVHTB 6 Q

TON 采用慢路由和快路由两种消息路由方式。在快路由中，可以使用 Merkle 证明直接中继消息，无需提交到中间的分片链中，避免了额外的延迟。但如果丢失收据，验证人不会受到惩罚，所以二者并行运行。

特点包括：节点只需知道邻节点的信息，无需知道其他节点的错误；基于一定规则而非纯异步收集输入消息；以及通过特殊交易显式删除已经被相邻的分片链传递的输出队列消息，防止了双重传递。

分片链的全局状态 - bag of cells 与垂直区块修复：

bag of cells是一组 cells，它们形成一个有向无环图（DAG）。更新它的过程是，将新状态表示为有其自身根的另一个“bag of cells”，移除旧版本中的所有 cell，然后联合新旧两组 cells 并删除旧的根。

在 TON 主链中，分片链中的每个区块实际上是一个小型区块链，也称为“垂直区块链”。如果需要修复错误的分片链区块，“垂直区块链”中会提交新的区块，这个新区块可以用来替换无效的“水平区块链”的区块，或者描述需要更改的区块之前版本的差异部分。当“垂直区块链”的增长速度超过原有的区块链，就可以替换成新的版本。

共识机制 - PoS 和 BFT 结合

POS - 网络中存在 3 个角色：

1. 验证节点：通过质押 TON 代币参与维护网络安全；

条件：需要质押 300 000 TON，同时满足硬件需求；

机制：所有区块由 100 ~ 1000 个选定的节点创建，节点每月选举一次，选举时封存提交的 TON Coin。在任职期间，选定的节点被划分成多个工作组，创建指定区块链上的新区块。每个新区块拿到工作组超过 2/3 的股份的节点的签名即算创建成功，若作恶则损失资金，剥夺资格；

2. Fisherman（通过指出别人的错误来获利）：通过发送无效证明检测验证节点是否尽职完成验证任务，如果验证节点通过无效证明那么验证节点则会受到惩罚；

3. 校对人（通过给验证人提名推荐某个节点成为验证器，若当选则获利）：它向验证人建议新的分片链候选区块，校对该分片链的状态和其他（通常是邻近的）分片链获取的数据，以及合适的 Merkle 证明来发送给验证节点。

BFT：相比较而言虽然 DPOS 更快，但是信任级别速度比 BFT 系统慢，因此 TON 选择 BFT。

与其他公链的比较

https://ton.org/en/analysis

TON 自身特点是区块 finality 时间迅速，交易时间快速，最大分片数量多且分片通讯速度高于以太坊：以太坊是所有用户都与一个智能合约交互，事务按顺序执行；而 TON 可以为每个用户钱包创建一条链，分片的并行计算、支持瞬间跨分片交流和 TVM 支持异步计算构成了其高 TPS 的理论基础；

TON 与 Solana 和以太坊之间的核心区别为资源支付和异步性带来的不同，优势是更高的可扩展性和灵活性，缺点是增加了应用开发和维护的复杂性。

1. 资源支付：在 TON 区块链中，每个智能合约都需要支付其自身的资源成本。即每个智能合约都需持有一定数量的 TON 代币，并使用这些代币来支付其运行所需的资源，如计算、存储和网络传输。

这种设计可以避免用户直接承担费用，但也意味着智能合约需要持有足够的 TON 代币来支付其运行成本。如果一个智能合约的 TON 代币余额耗尽，该合约将最终被删除。这是一种自动清理机制，可以避免区块链上的数据膨胀

2. 异步性：在 TON 区块链中，智能合约之间的调用是异步的，而不是原子的。这意味着当一个智能合约调用另一个智能合约时，该调用不会立即执行，而是在交易结束后的某个未来区块上被处理。

4.TON 原生组件

TON P2P network

简介：是类似 IPFS 的 P2P 网络传输协议，可以访问 TON 区块链、发送交易，接收客户端账户或智能合约信息；

网络结构：TON 利用 Kademlia Distributed Hash Table (DHT)定位网络中的其他节点，节点之间使用 Abstract Datagram Network Layer (ADNL)进行通信，TON 还为每个 Shardchain 创建特定的 Overlay 子网络，确保有效的通信。

1. DHT：定位网络中的其他节点，当客户端需要提交交易至某 Shardchain 的 Validator，就可以通过 Key 在 DHT 中查找，获取 Validator 的位置；

2. ADNL：ADNL 对传统的 TCP/IP 分层架构中的网络层进行了抽象。为了便于识别身份，节点之间不需要考虑 IP 地址，而是利用抽象网络地址（Abstract Network Address）进行通信。该地址是一个基于 ECC 公钥以及其他参数的哈希值的 256 位整数，为不同 Shardchain 之间的分割提供了基础；

3. Overlay：不同的 Shardchain 无需、也无能力处理其他 Shardchain 上的交易，所以需要在 TON Network 内部为每个 Shardchain 构建并 Overlay 子网络，Overlay 网络内部通过基于 ADNL 的 gossip 协议进行通信。

TON DNS

简介：基于 TON 的分布式可扩展的命名系统，类似 ENS 。 TNS 的可将可读的名称（如“alice.ton”）解析为计算机可以识别的标识符。目前可在 https://fragment.com/ 中出售/拍卖（已售出的 TNS 中最贵的是.news，售价达 994000 TON，约 2 m 美金）；

发布时间： 2022 年 7 月；

可用程度：完全可用。用户可以购买用作昵称和钱包地址，开发人员可在开发时用.ton 域名取代智能合约地址；

生态支持：目前 Tonkeeper、TON Web Wallet 和 Tonscan 服务已经集成了对 TON DNS 的支持，可在 GeTelegramems 或 Disintar 等 NFT 市场进行出售。拥有域名的用户每年需支付 0.015 TON 来继续租赁域名。

TON storage

简介：采用 BT（文件种子）技术的 P2P 文件共享与储存方案，类似 dropbox。可用于存储 TON 区块链的区块副本以及状态快照，官方称其为“去中心化的 Amazon S 3 ”；

发布时间： 2023 年 1 月；

可用程度：用户可通过 demo 界面 存储 50 mb 以下的文件，但需要自己创建 Bag（通过 TON Storage 分发的文件集合）并获取 Bagid 部署智能合约，使用较复杂。开发者可通过部署合约的方式在 TON 上存储和分发任何大小的文件或在 TON storage 中托管静态 TON 站点；

发展：未来 TON Storage 客户可以使用存储节点注册表查找满足其需求的存储节点，且会进一步优化用户端体验，推出类似于 Dropbox 的 TON Torrents 应用和 TON Storage API。

TON Proxy

简介：Proxy 是一种特殊的网络服务，允许一个终端通过此服务与另一个终端进行非直接的连接，进而提供对网络的匿名访问。TON Proxy 是 TON 网络代理层，用于隐藏 TON 节点的 IP 地址。用户可以直接在设备上启动入口代理，未来该部分将以简单 DAPP 的形式提供给所有用户；

发布时间： 2022 年 10 月；

可用程度：用户可以运行不依赖于固定 IP 的托管在 TON Storage 上的.ton 网站，并可以使用与 HTTP 代理兼容的公共入口节点访问这些站点；

发展：未来主要的 TON 服务都可集成 TON Proxy，类似匿名 I 2 P 网络的功能，将确保 IP 地址的安全，用户也可以运行中间代理节点并获得收益。

TON payments

简介：集成微支付通道的小额支付平台，类似“闪电网络”。能用于链下交易和支付 TON Services 的服务费；

可用程度：目前 telegram 集成了“Wallet”Bot 服务，允许用户与零售业务之间进行虚拟货币支付交易，直接在 Telegram 界面内搜索 @wallet 即可进行支付、购买加密货币、P2P交易等，只有频道的创建和关闭需要支付网络费用。接下来将引入商家市场，在@wallet 界面投放产品和服务；

发展：未来会搭建多节点网络，用户可以通过中间节点进行小额支付，而无需创建新渠道，中间节点可以收取佣金。

TON space

简介：Telegram 中的自托管钱包。@Wallet 钱包是 KYC 的，只能调用支付，TON space 是用户自己保管私钥，所有智能合约都能调用；

发布时间： 2023 年 9 月；

可用程度：目前是 beta 状态，可在个人账户中的 wallet 中激活，仅有基础的收付款功能。未来 TON Space 将充当区块链账户，全面支持包括 Toncoin、Jettons 和 Collectibles 在内的 TON 生态系统

Telegram Apps Center

简介：提供展示 Telegram 机器人和网络应用程序 (TWA) 生态系统的目录；

发布时间： 2023 年 8 月；

可用程度：目前有三种类型的项目，分别是机器人、游戏和 Web 应用程序。用户可以通过官网、Telegram bot 和Telegram Channel about Trending Apps 访问 TWA。开发者也可以在专用应用程序审核机器人的帮助下，无缝提交开发的新程序。

5.Telegram 生态和 Ton 生态

Telegram 生态现状： Bots+API+钱包托管

Unibot 的暴涨和收入的能力带火了低风险高收益的 Bot 赛道：Telegram Bot 创建/开发简单，与 telegram 绑定，使用便利，其收入来自交易额和税费，在熊市的的强盈利能力（第一梯队的$unibot $MEVFree 等项目每月最高收入在 500 万美金以上，第二梯队的市值不到 1000 万美金的 Bot 项目月收益也有几十万美金），促使更多团队转向 Bot；

目前 Telegram 生态以「Bots+API+钱包托管」形态为主：Ton Bot 生态目前来看较难直接复制微信小程序，微信平台申请较为容易，但 TON 的小项目 build in 需要复杂的申请流程，导致想嵌入体验更好的官方钱包很难，同时开发难度和成本高 + 很难获得 grant 扶持是小项目的部署现状。因此小游戏会选择折中的“游戏主体在以太坊 layer 2 + telegram 的 API + 链下计算”的快速部署方式。

为什么 Telegram 需要 TON 生态？

尽管 Bot 数据好，但对 TON 公链没有实际支持：Bot 生态未用到 TON 生态组件，也未映射代币；

TON 是 Telegram 商业化的最好的途径之一：由于历史敏感性 + telegram 的盈利方式单一 ，Telegram 必须选择更适合的方式商业化，Telegram 创始人在创立初期就对区块链有所布局，现在通过 Bot 的引流让 TON 很大可能成为一条独特的应用 base 公链。因此 TON 就是 telegram 的最好的途径之一；

为什么开发者需要到转到 TON 生态？

TON 基金会生态发展抓手包括：Apps Center 倒流、TON 原生支付、telegram 社群推广、基金会 Grant 和投资、解除 API 限制等；

1. TON 原生支付：当前@Wallet 钱包是 KYC 的，只能调用支付，功能单一且拓展性差，产品想扩大应用规模和功能性就必须有基于公链的原生支付组件。而 TON space 就是面向 web3 用户的自托管钱包，能调用智能合约，且能帮助项目方，不需要让用户导入钱包即可使用支付，产品想象度大大提升；

2. telegram 社群推广：

平台：有[官网](https://tapps.center \h )、手机端推广 bot 和推广 Telegram 频道；

GitHub：完成赏金任务的开发者会在 TON 的 GitHub 中以名人堂的形式展现，目前有 106 个，项目涵盖代码拓展、NFT 和开发教程等；

广告：Telegram 的赞助信息可以显示在大型公共 1 对多频道上，让每个在 telegram 上观看特定频道的用户都会看到相同的赞助信息，无具体用户分析，是广撒网式投放方式。TON 基金会有权力投放部分项目广告。如目前 gamble 项目 @Whale 在 TON APP 官网投放主页横幅广告，除此之外还有类别广告和搜索广告可以投放；

3. 基金会 Grant 和投资：grant 最高 5 万美金，无具体标准；投资基金有盈利需求，最高 10 万美金。

Grant： 23 年 10 月，TON grant 计划批准了 10 个项目，包括 AI 智能模型 Tali AI、Ledger Stax 集成应用 Ledger APP for TON、免许可 Launchpad TonUP、去中心化交易工具 Optus DEX Aggregator、P2E 游戏 Gatto、TON 市场调研工具 QSTN、多签钱包 Tonkey、应用市场 ton.app、综合通信协议 TON Connect 2.0、追踪平台 TON Tracker；

基金会：目前基金会已经发出 $ 1200, 000 的 grant，资助了 100+ 项目，平均每个项目一万美金。23 年基金会拨款项目有 TON 钱包与 KaiOS 的集成、借贷协议 EVAA Protocol、应用平台、开发组件、预测市场、launch 平台、与 AI 的集成等。

4. 解除 API 限制：Telegram 对 bot 每秒钟能发送多少条信息进行了限制（bot 发送文件最大容量为 2000 MB，将请求转换为 HTTP 的 Bot API 服务器的文件大小限制为 20-50 MB），而 TON 可以使用不受限制的匿名 RPC 访问 API，支持数据索引和获取等，开发灵活性大幅增加；

5. 稳定的平台：Bots 很难为用户解决支付和出入金，但 TON 生态应用可以无缝衔接对应的基础设施；Bots 会在 DeFi 上加收手续费，而 TON DeFi 应用只有一层更低的收费。

6. 更好的用户使用体验：Bots 响应速度受限，交互方式单一且多交互并行能力差；但 TON App 能更好解决这个问题。当前 Telegram Bots 的交互形式更像是一个单调的代理页面，而 TON 上的项目是一个功能交互性、可组合性更强的 APP，并且上了 APP Center 的项目都可以被@Wallet 给连接起来，因此 TON 与 Telegram Bots 的项目的纵深差别非常大。

4. CEX：目前 TON 上有 44 个 CEX，如 OKX、火币、bit、coinEX、gate、biTelegramet、moonpay 等，剩下的是一些小型/本地化交易所，如西班牙的 bit 2 me 和伊朗的 اوکی اکسچنج；

5. 跨链桥：官方桥当前仅支持 BSC 与 ETH 和 TON 之间的资产互通，转入转出资产仅限制为 TON。第三方桥 Orbit Bridge 在 22 年 12 月 1 日宣布支持 TON 网络。

除了原生资产 TON 外，Orbit Bridge 支持 Ethereum、BNB、Klayth、Polygon 等多个链涵盖 USDC、USDT、MATIC 等多个资产在内的在 TON 区块链间的相互跨链。

6. utilities：以 Bot 为主，提供类似小组件的辅助功能，如水龙头、通知机器人、交易追踪、社区管理工具和有奖问答等；

7. NFT：共有约 50 个系列。较知名的系列有 Annihilation（地板价 198 TON，volume 177.2 K TON）、CALLIGRAFUTURISM（地板价 650 TON，volume 21.6 K TON）、OCTOPUS BOYZ（地板价 87 TON，volume 75.8 K TON）和 METAMORPHOSES（地板价 99.5 TON，volume 38.9 K TON），无知名艺术家入驻，volume 一般；

8. game：总计 30+，热门的两个 gamble 项目是集成平台，且以 bot 形式嵌入 telegram。其他游戏项目看起来略土狗，基本都是小程序直接启动，无官网；

9. DEV 组件：目前共有 7 个组件，包括 IntelliJ Platform（开发者工具的开源平台） 的 TON 开发插件、Tonana 跨链桥、将合约转换为二维码、Ton 的合约验证等，除此之外无其他有用的组件，甚至会出现官网打不开的情况，整体开发者组件生态较一般。

6.投资展望

看好官方扶持的基础设施

TON 网络：Telegram 的 MTProto 协议可以很容易地嵌入到 RLDP 协议中，从而能将 Telegram 转换为 ton-service，这有助于 Telegram 的转型，同时其他同类型消息和社交网络服务也可能从中受益；

TON 原始组件/中间件：当前已有基本的网络组件（P2P 和 proxy）、社交（DNS、storage）、payment（wallet 和 TON space）。未来可以围绕基建进行组合和拓展，如一站式网络部署 infra，动态 site 的搜索引擎，市场数据类 dashboard；defi 类也会有 DEX 和 aggregator 的机会；考虑到 TON 上游戏类项目较多，基于 ERC 721 的 NFT market 也有一定潜力。

看好小程序应用

思路 - TON 与以太坊的生态，着力点和开发者生态都不一样：以太坊重 defi，crypto 叙事强；TON 重视小程序的多样性和广度，叙事偏向 web 2.5 ，体验性大于 crypto 正统性，就是想达到 mass adoption，语言也对老 web2 开发者（偏好 C 语言）更友好。

TON 的应用侧机会更多，官方重点扶持小程序，纯 crypto 应用层很难走通，优先级是先小程序后 defi。crypto 的技术创新度不是第一性，稳定和安全是重点。目前 Telegram 的用户习惯还没办法对标微信，生态结构、用户粘度、用户心理预期更像 qq，很多 functional 的功能为主。

Ton space 和 telegtam 入口端能吸引大量web2流量，整体生态有流量来源、应用场景、社交关系、支付网关和用户触达的需求，前期组件可参考微信小程序发展形式，针对某一类垂直领域：

1. 流量来源：DApp 任务分发平台；Telegram Ads 目前是广撒网式投放，若有针对性投放业务会更受欢迎；其他 Web 3 项目的入驻（如 Readon）；

2. 应用场景：App Bot 和 Web App；Web 2 移植内容（如 TON 正在和网易官方团队的卡牌游戏 card ahoy 沟通合作；电商类转 Web 3 ，如虚拟电商、现实世界电商，甚至 crypto 电商等；

3. 社交关系：基于 TNS 的社交图谱协议；与游戏结合的社交/交易市场；分享和邀请平台；

4. 支付网关：信用卡和加密货币的结合；与 RWA 的结合；类似“余额宝”的稳定挖矿平台；

5. 用户触达：Telegram channel；未来可通过 TON space 和 @wallet 的联合触达更多 Web 2 转 Web 3 的用户。

谨慎看好 DeFi、MEV 和 ZK 

1. DeFi：此类应用无需有非常强的 crypto 创新度，仅需满足基本的质押、swap 和交易功能即可，对安全性和使用稳定度有较高要求，可能是有传统 crypto 积淀的大交易所/产品的机会；

2. MEV 问题：目前无大额交易，暂未到解决此类问题的时候，当前首要任务是要“造”更多资产，增长交易需求；

3. ZK 和 bridge：二者都很重要，但是现在部署尚早，需等基建继续搭建。且 TON 成长关键不在 bridge，因为 bridge 是存量分配从以太坊生态吸引流动性，而 TON 想开拓新流量 ，法币稳定币是更重要的关注点；短期内不看好 Layer 2 和 zk 方向；

其他赛道展望

1. Dev Tools/Community：TON 上 Dev 的特点是部署和开发难，但是理论的 TPS 更高。官方技术资源较少，对于小团队的开发而言有难度，因此首要方向是布局开发者生态，优化开发者体验。此类方向有 TWA 接 RPC，或开发工具包、无代码部署平台等业务方向，但考虑到 TON 有 Archive node 的要求，此类项目对于团队技术沉淀和背景要求较高；

2. Social Payment 及并行支付基础设施：如创作者经济类的 Onlyfans，需解决创作者和用户交互的问题；同时还有私域流量变现等针对社媒运营者的方向；

3. 游戏与社交图谱的绑定：如社媒流量分发平台、小游戏平台和社交图谱的绑定以及 NFT 市场。或传统游戏的移植，如类似天天快跑、斗地主之类的休闲游戏、开心农场类的社交游戏，这些都天然能与代币经济模型结合在一起；

4. 新价值创造与流通：与传统社媒的连接。

最后

自 2018 年创立以来，TON 经历了一系列的挑战和转型，包括 ICO、重启和生态系统的重建等。在基金会的积极推动下，其市值实现了显著增长，用户基础也在扩大。具有灵活无限分片技术和高效的共识机制，使其成为一个具有扩展性的、安全性高的区块链网络。
对于 TON 的未来，预计在 Telegram 的扶持下，它有望扩大市场份额，实现更大的用户覆盖。TON 的优越性和 Telegram 支付的整合，使其在社交媒体生态中有很大的发展空间。但是由于 TON 的开发语言以及异步设计的复杂性，对开发者来说挑战也很大，这可能限制了生态系统的快速发展。
TON 生态已经收录 551 个应用，并且这个数字还在快速增长，表明 TON 正逐渐构建一个多样化和丰富的生态。未来通过优化基础设施、提供一站式部署服务、提升各项功能体验等方式，TON 生态可能引入更多开发者参与进来。
整体看来，TON 有着良好的发展趋势，其整合了 Telegram 的用户基础，以及良好技术框架，是其具有巨大的发展潜力。TON 的基础设施、小程序应用等多个方向值得关注，但无论投资者还是开发者，都需要在深入理解 TON 生态和相关技术的基础上，谨慎做出参与的决策。
